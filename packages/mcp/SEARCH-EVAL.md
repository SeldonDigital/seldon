# Semantic Search Eval — Quality Gate & Model Selection

Status: **gate met (96.5% ≥ 90%). Human authorship pass complete
2026-07-12** — reviewed via eval-review.html; found and fixed the
`carbon-search` tag gap (icon-tags-overrides.ts) and an upstream Core
icon-rendering bug (reported separately, unrelated to search).

## Setup

- Eval set: [`src/search-eval-set.ts`](src/search-eval-set.ts) — 57 authored
  query→expected pairs (34 icons, 20 components, 3 themes/fonts; gate
  requires ≥40 total, ≥25 icons, ≥15 components). A pair scores a hit when
  **any** of its acceptable ids appears in the top 8 — the icon catalog
  carries many near-duplicates across its four sets (carbon, lucide,
  material, seldon), and any of them serves the intent.
- Harness: `npm run eval:search` ([`scripts/search-eval.ts`](scripts/search-eval.ts)).
  Embeds the full 8,391-entry corpus per candidate model, embeds queries
  one at a time (the server's runtime shape — batching perturbs pooled
  vectors by ~0.007 cosine, enough to reshuffle a tight top-8), and scores
  through the **real** search path: `searchCatalogIndex` + calibrated
  semantic blend + the tool's kind-diversity slot.
- Icon entries carry **vendored upstream search tags**
  ([`src/icon-tags.ts`](src/icon-tags.ts), generated by
  `npm run build:icon-tags` from carbon aliases / material symbols tags /
  lucide tags — 7,975 of 8,318 icons covered), plus a small **hand-curated
  overrides file** ([`src/icon-tags-overrides.ts`](src/icon-tags-overrides.ts))
  for icons whose upstream data is too sparse to find by a natural query —
  the durable home for fixes found during the review (e.g.
  `carbon-search` had zero glyph words in its own Carbon aliases). Overrides
  merge on top of, never replace, the vendored tags, and survive re-running
  `build:icon-tags` (which fully regenerates the vendored file). Both feed
  keyword `tagWords` and the embedding passage text.
- The cosine noise floor (`simFloor`) is grid-searched per model
  (0.0125 steps through the peak region); the winner's floor is recorded in
  [`scripts/embedding-models.ts`](scripts/embedding-models.ts) and stamped
  into the index by `npm run build:search-index`.

Earlier full-candidate sweep (pre-tags, same harness): all-MiniLM-L6-v2

**Decision: `Xenova/bge-small-en-v1.5`, q8 ONNX weights** (~34 MB
vendored, 384 dims, query prefix "Represent this sentence for searching
relevant passages: "). fp32 scores one pair higher (94.7%) at ~4× the model
size and slower query embedding; q8 clears the gate with margin, so small
wins. Swap by editing `DEFAULT_MODEL_ID`/dtype and rebuilding if that
trade ever flips.

## What moved the number (design notes)

1. **Vendored icon tags** (biggest single win: keyword floor 56.1→75.4%,
   hybrid 91.2→93.0%): the upstream icon projects' own search aliases give
   icons the concept vocabulary their labels lack — carbon `close` ←
   "x, dismiss", carbon `view` ← "eye, eyeball", material `autorenew` ←
   "circular arrow". Generic material tags ("ui", "app", "button", …) are
   stripped at generation; tags are capped at 12 per icon.
2. **Query stopword filtering** (keyword path): "x **to** dismiss" no longer
   lets every "Move To" icon score on "to".
3. **Per-query relative calibration** (semantic path): cosines normalize
   against the query's best match above the absolute noise floor; absolute
   cosine bands are too narrow and query-dependent to threshold directly.
4. **Kind-scoped normalization**: kind-filtered searches compete within the
   searched kind, not against 8k icons.
5. **Kind-diversity slot**: kind-unfiltered searches keep each kind's *best*
   match on the page. (A first version accidentally seated each kind's
   *worst* match — `Map` keeps the last duplicate key — costing three pairs;
   regression-tested now.)
6. **Solo query embedding** at eval time to match the runtime exactly.
7. **Synonym curation** (the zero-result log's intended loop): funnel→filter,
   marker→pin/location, photograph→image, tick→check, etc.
8. **Diversity quota raised 1→2 per kind** (91.2%→93.0%'s successor,
   93.0%→96.5%): the original rule guaranteed exactly one seat per kind
   *absent* from the raw top 8, so a query with two valid answers of the
   same kind (`nav`+`bar`, or two components both matching "photograph
   placeholder") could only ever surface one, however well the second one
   scored. Generalized to a per-kind quota (default 2): seats are now filled
   by each kind's best *remaining* candidates, evicting kind-*surplus* raw
   picks first so a same-kind injection never displaces the seat it's
   supposed to be adding to (an early version had exactly that bug — fixed,
   regression-tested).
9. **Per-icon tag overrides** ([`icon-tags-overrides.ts`](src/icon-tags-overrides.ts),
   review-sourced): patches individual icons whose *own* upstream
   alias data is sparse even when a sibling in another set is fine —
   `carbon-search`'s Carbon aliases are function-only (`[find, investigate,
   explore, look]`, no glyph words), so "magnifying glass" couldn't find it
   even though `lucide-search` already carried the pair. Doesn't move
   pair-level recall (the pair was already a hit), but fixes entry-level
   findability — matters if a user ever needs the Carbon-specific icon
   rather than "any search icon."

   **Known partial gap, deferred:** the same override did not fix
   `material-search` (still rank ~12, not top-8) — adding its missing
   glyph tags only brings it to the same 0.700 tag-match tier as ~7 other
   icons already there, and it loses the alphabetical tiebreak among them.
   Fixing this needs either a finer-grained tag-match score (so a 2-word
   exact match outranks a 1-word one) or fewer ties at that tier — a small
   scoring change, not another tag addition. Left for later since the pair
   already resolves via `carbon-search`/`lucide-search`.

## Calibration sensitivity

The pre-tags result sat on a one-grid-step peak (91.2% at 0.5875, 89.5%
either side). With icon tags the peak is broader and the floor lower
(0.55); still, re-run the grid whenever the eval set or corpus text changes
— it's cheap (models cache in `.cache/`).

## Reproducing

```bash
# refresh vendored icon tags (network; regenerates src/icon-tags.ts)
# hand edits go in src/icon-tags-overrides.ts instead — never touched by this
npm run build:icon-tags

# compare candidate models (downloads into packages/mcp/.cache/)
npm run eval:search [-- --model Xenova/bge-small-en-v1.5]

# build the shipped index + vendor the model (search-index/, gitignored)
# rerun after ANY tag change (vendored or override) — embeddingText changed
npm run build:search-index

# regenerate the review UI (eval-review.html, gitignored) against the
# current shipped index — rerun whenever the eval set or search changes
npm run build:eval-review
```

The server needs no build step to fall back: without `search-index/` or the
optional `@huggingface/transformers` dependency, `search_catalog` is
keyword-only, silently — now at the 75.4% floor. Index/model
mismatches fail loudly to stderr and fall back.
